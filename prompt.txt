-----       Prompt       -----
Your task is to build a professional, user-friendly website to manage and monitor an existing deep learning pipeline (using CNNs and Transformers) that runs batches as subprocesses.
I have provided the code files for the existing pipeline.

Requirements:
1. Website functionalities:
   - Start, stop, resume, and terminate subprocess batches.
   - View logs, errors, notifications, and status of all batches.
   - Access output files (plots, CSVs, TXT files) for a specific batch stored in its corresponding subdirectory.
   - View batch metadata: start time, elapsed time, models used, controls (augmentations, pretraining), configs, and any skipped experiments.
   - Receive notifications for skipped experiments, errors, or completed batches.
   - Change the config and controls of any batch directly from the website (e.g., models, augmentations, pretraining, hyperparameters, etc.).
   - Create and add new batches:
       * Users can select models from a user-friendly input dropdown or input a list of model names as text.
       * Edit configs and controls for each batch; defaults should already be present.
       * Option to restart all batches (using a default segregation of models into batches).
2. Batch management:
   - Subprocesses respect existing logic: if a batch is already completed (results stored in master output), it should skip processing (it does currently) but notify the user.
   - Resuming a batch should correctly restart from the whole suprocess where it left off, preserving consistency with master results.
   - All results must be stored under a single root output directory, with subdirectories for each batch.
3. Notification system:
   - Dedicated notification center showing all notifications in the UI.
   - Notifications stored on disk in a performant format (e.g., lightweight database, JSON lines, or SQLite) to survive server restarts.
4. Performance:
   - Website operations should minimally affect GPU/CPU resources for batch processing.
   - Real-time updates and monitoring should be responsive but lightweight.
5. Backend:
   - Preferred: FastAPI for API endpoints and subprocess control.
   - Alternative: any more performant backend technologies if needed.
6. Frontend:
   - Interactive dashboard to show all batches with status, logs, elapsed time, metadata, and outputs.
   - Easy browsing and download of batch output files.
   - Notifications/alerts for errors, skipped experiments, or completions.
   - Clear UI for quickly identifying running, completed, or errored batches.
   - Controls to edit batch configs and parameters.
   - User-friendly batch creation interface with dropdown or text input for models.
7. Optional but valuable, ensure minimal effect on GPU/CPU resources for the actual batch processing:
   - Filtering and searching batches by models, configs, or time.
   - Graphical visualization of batch progress or performance metrics.

Deliverables:
- Full FastAPI (or your preferred technology) backend implementation with endpoints to control, monitor, configure, and create batches.
- Frontend dashboard (can use React, Vue, Tailwind with minimal overhead, or any other performant frontend technology you prefer) with interactive controls, file viewing, notification center, and batch creation UI.
- Clear structure for batch outputs, logs, notifications, and live config changes.
- Integration with existing ML pipeline code without interfering with GPU-intensive computations.
- Instructions for running the web app alongside existing batch processes.

Please generate the full code structure including backend, frontend, and integration logic, with emphasis on efficiency, clarity, configurability, and delightful user experience.
